0 · "Is this just a wrapper that costs more in API calls?"

The honest answer: for a single one-shot question, a frontier chat LLM with web search is genuinely close to good-enough — and Scout costs marginally more for that one query. So Scout must not position as "a nicer way to ask once." Its reason to exist is the set of things a stateless chat model structurally cannot do.

Where the wrapper critique is RIGHT

  • • One-shot "best X under $Y" → ChatGPT/Gemini/Google AI Mode answer fine.
  • • For that single call, Scout pays more (LLM + search + SERP).
  • • "It searched the web and summarized" is not a moat — everyone has that.

⇒ Don't sell the one-shot. It's a losing fight against free frontier chat.

What chat models STRUCTURALLY can't do (the moat)

  • 🔔 Watch & notify over time. Stateless models can't monitor price/stock and alert you. Needs a persistent backend — a real product.
  • 💸 Verifiable live pricing. Chat hallucinates prices & stale stock. Scout grounds "cheapest now" in a Shopping API + enforced citations.
  • 🛡️ Auditable trust layer. Chat launders sources into one voice; Scout flags sponsorship/bias and surfaces dissent (§2).
  • 🗃️ Accumulating data. Canonical catalog + price history + credibility compound; a chat thread is ephemeral.
The real incumbent isn't ChatGPT — it's Google. Google Shopping + AI Overviews + price tracking already does much of this. Scout's wedge vs. Google: cross-retailer neutrality (ads are Google's business), an explicit trust/bias layer, and multi-source depth. Defensible, but Google is the gorilla. (This is grill question G6.)

2 · Detecting bias / sponsorship / bots, and where we search

How we detect bad sources

🏷️ Sponsorship / conflict: markers (#ad, "gifted"), affiliate-link density, brand-owned domains, thin affiliate farms. LLM classifies using these features.

🤖 Bot / fake review: review-burst patterns, templated language, unnatural rating distributions, generic praise; Reddit account-age/karma where available.

🎯 Irrelevant / stale: semantic relevance to model + criteria; recency; wrong-variant detection.

⚖️ Credibility score: independence + original-testing + cross-source corroboration → a per-source score that weights synthesis and is shown.

👁 Surface, don't suppress: flagged sources shown with the flag; legit dissent surfaced.

Honest: perfect detection is impossible — value is being systematic & transparent. (This is grill question G3.)

How we decide where to search

🧭 Tiered taxonomy per query:

  1. Live offers → Shopping SERP.
  2. Expert reviews → web search, quality allowlist + open discovery.
  3. Community/forums → web search hitting Reddit (+ optional API).
  4. Video → YouTube API.
  5. Retailer ratings / Q&A.

🎯 Targeted per-criterion queries, not one generic search.

✅ Require ≥N independent sources per claim; explicitly seek a dissenting source.

G1 · resolved

Generalist stays — but with calibrated confidence, not blind uniformity

You pushed back: generalist is the whole point, even personally — and you're right that narrowing the product would cripple your own non-electronics use, while the architecture is identical either way. I withdraw "narrow the scope." But your "what causes the dip?" question deserves a real answer, because the dip is real — just not in the code.

Worked example: headphones vs. mattresses (why the dip is real)

Same pipeline, two queries. The difference isn't the code — it's the category's source ecosystem + how dedup-able it is:

"ANC headphones <$300""mattress, side sleeper <$1200"
Independent ground truthRTINGS lab-measures ANC; r/headphones is expert"best mattress 2026" web is ~all affiliate-farm SEO; little real testing
Trust-engine signalcalibrate sponsorship vs a known-independent baselinedown-weight affiliate sources → little left; weak signal
Dedup / "cheapest"stable model # + GTIN across retailers → cheapest trustworthyretailer-exclusive variants, firmness/size combos, bundles → "cheapest" easily wrong

So a generalist v1 doesn't crash on mattresses — it produces a confident-looking report built on spam with a shaky "cheapest," and you'd never know it was weaker than the headphone report. That silent unevenness is the actual risk.

Shared resolution (better than either opening position): stay generalist at runtime (ask about anything), but (1) make the trust engine emit a per-report confidence signal and say so when a category is affiliate-contaminated or un-dedup-able ("limited independent sources — treat with caution") instead of faking confidence; and (2) build eval golden-sets depth-first, leading with categories where ground truth exists, so we know where Scout is strong vs weak. Adds a confidence field to the report model + the confidence chip now in the §1 mock.

G5 · discussion

Where AWS/CDK earns its keep — and what each option actually costs

You're right to keep most things third-party (Vercel, Supabase, Serper, OpenRouter/Anthropic) and to want a deliberate place to learn AWS/CDK. Here's my concrete opinion on the where, then the costs you asked for.

My opinion — put AWS exactly at the orchestration layer. The single best place to swap in AWS+CDK is the async pipeline + watch cron: Step Functions (the research state machine) + Lambda (per stage) + EventBridge Scheduler (watch cron), provisioned with CDK in TypeScript (same language as the app — strong résumé signal). Why this seam and not others: (1) it's already behind a clean boundary (it only reads/writes Postgres and calls the Source Adapters), so swapping Inngest→Step Functions doesn't entangle Vercel or Supabase; (2) it's a genuine fit — a multi-step durable workflow + scheduled jobs is the textbook Step Functions use case; (3) it's a self-contained, portfolio-worthy infra project. Keep everything else third-party. Adjacent AWS bits you'd naturally learn next: S3 (report/offer cache artifacts), SQS (queues), CloudWatch (observability).

The honest trade: Step Functions + Lambda + CDK is materially more work than Inngest (Inngest is ~zero-config durable steps inside your Next.js app; AWS is ASL state machines + Lambda packaging + IaC). You're buying learning, not speed. And a gotcha: Lambda bills for wall-clock while a step waits on the LLM — fine & cheap at small scale, but for long Thorough steps use Step Functions async/callback patterns (or a Fargate task) rather than a Lambda that sits idle on the model.

G6 · resolved

No technical moat; defensibility = independence/trust + data + Google's ad blind spot (revisit seriously later)

Accepted for v1, flagged to revisit before any commercial push. The honest position: anyone can call the same APIs, so there's no tech moat — the defensible angles are positioning ones (Google structurally won't flag sponsored/affiliate results because that's its revenue; the Wirecutter/Consumer-Reports independent-trust play; accumulated watchlists/data + switching cost). ＋ your question

"Why not just use Google's APIs?"

We actually do — heavily. Our price backbone is Google Shopping (Serper.dev returns Google Shopping results); Gemini is a candidate model in the cascade; and YouTube (Google-owned) is a review source. The catch is there's no single turnkey "Google API" that delivers Scout:

  • 🛒 Shopping Content/Merchant API is for merchants to upload their own products, not for third parties to query the whole cross-retailer catalog — which is exactly why SERP providers (Serper/SerpApi) exist to read the public Shopping results.
  • 🔎 Custom Search / Programmable Search is capped (100/day free, 10k/day) and site-restricted — not full open web research.
  • 🤖 Gemini + Search grounding returns a grounded answer, not the structured per-source evidence + offer data Scout's trust layer needs — and it's a black box you can't audit or re-weight.
  • 🛡️ No trust/bias layer, no price-watch/history API — those are exactly the parts Scout builds regardless.
The deliberate reason not to build entirely on Google: if Scout's whole value is neutrality, we want Google's raw data (Shopping offers) but not Google's ranked/ad-influenced answer as our output — handing Gemini's grounded answer straight to the user would import Google's incentives into Scout. We take Google as a commodity input behind our adapters, and do the independent multi-source synthesis + trust scoring ourselves. The adapter design also means we're never locked to it.
G7 · active (last)

What does "v1 works" actually mean — the success / eval bar?

Everything above leaned on "we'll measure quality." This is where we make that concrete, before building, so we don't move goalposts. The core bet — repeated all session — is "a Scout report beats ~20–30 min of your own Googling." Let's define how we'll know.

My recommended v1 success bar: (1) Quality vs. baseline — on a golden set of ~15–20 of your own real purchase questions across a few categories, you (blind-)rate the report ≥ as good as 20–30 min of your own research in ≥75% of cases. (2) Citation integrity — 100% of cited URLs resolve and support the claim (automated; zero fabrication). (3) Price correctness — the "✓ cheapest" badge is never wrong on spot-checks (ties to G2). (4) Trust signal works — on a labeled fixture set, ≥90% precision flagging explicit-disclosure sponsorship, and dissent surfaced when it exists (ties to G3). (5) Latency — Quick path streams first content < ~10s, completes < ~45s. (6) The real signal — you actually reach for Scout for your own purchases over a month instead of Googling. The eval harness for (1)–(4) is built in Phase 0, not bolted on later.
